source = DataFrame(np.random.uniform(100, 200, [60,30]))
cols = ['price', 'volume', 'open_interest']
scale = (len(source.columns) / len(cols))
source.columns = [scale  * cols]
sym = lambda x: np.random.choice([abc for abc in x], 
                                 np.random.choice([2,3]))
month = lambda x: np.random.choice([abc for abc in x], 
                                 np.random.choice([1]))

contracts = np.ravel([[(''.join(month(string.letters[:26])) + 
             str(np.random.choice([14,15,16])))] * len(cols)
             for x in xrange(len(source.columns) / len(cols)/ 2)])

level_1 = len(source.columns) / len(contracts) * list(contracts)

numsyms = len(source.columns) / (len(set(level_1)) * len(cols))
underlyings = [''.join(sym(string.letters[:26])) for x in xrange(numsyms)]
level_0 = np.ravel([[sym] * len(set(level_1)) * len(cols) for sym in underlyings])

source.columns = pd.MultiIndex.from_tuples(zip(level_0, level_1, source.columns))
source.index = pd.date_range(start= dt.datetime.utcnow() - dt.timedelta(days=len(source.index)-1), 
                             end = dt.datetime.utcnow(), freq='D')

Rolling Positions

Since we are dealing with actual futures contracts versus synthetic continous timeseries like lesser systems, a methodology needs to be inplace to ensure that the algorithm doesn't buy underlyings at numerous expirations. Thus the need for a roll method, ultimately this filters data available to the algorithm.

from alephnull.roll_method import roll
futdata = FuturesDataFrameSource(source.tz_localize('UTC'))

#Here I define my front month as that with the greatest open interest
my_logic = lambda x: x[x['open_interest'] == x['open_interest'].max()]
class FrontTrader(TradingAlgorithm):

    @roll(my_logic)
    def handle_data(self, data):
        [self.order((sym, data[sym]['contract']), 2) for sym in data.keys()]
        return data

Issues

  • Roll Return Type

    Currently roll returns a dictionary instead of a BarData object

  • Multiple Front Months

    Behind the scenes roll converts the Bar into a DataFrame and applies a groupby operation using your logic, this allows you to define your front month as to your liking and can be as simple as a lambda function. The problem is that the logic can result in multiple contract returns which results in various issues, A standard methodology needs employed to resolve multiple contract returns.

  • Uses Position Amount to Offset

    When switching to the next contract, the roll method looks at the aggreagate position sizes for each underlying, it then sells off the current holdings and buys into the front month. The problem being that orders dont fill until the next bar, and large orders may fill over several bars. Therefore after rolling, outstanding orders for a prior contract aren't included in the amount directed towards the front month.

  • Outstanding Orders

    As previously mentioned, it is undesireable for outstanding orders to exist after roll associates your strategy with a new front month. Thus on roll over all outstanding orders need to be cancelled.

    • There is also some arithmetic error here as noticed in the position sizes

    Handling Spreads

    API

    Eventually I hope add methods to BarData/SIDData To facillitate spread trading, the API would be similar to the following 

    class SpreadTrader(TradingAlgorithm):

    #defines front month to be contract with greatest OI
    @roll(lambda 
    x: x[x['open_interest'] == x['open_interest'].max()])
    def handle_data(self, data):
        for sym in data.keys():
            #no need to iterate through contracts as they
            #have already been filtered by the roll method
            
            #purchases 5 contracts of the front month
            self.order((sym, data[sym].contract), 5)
            
            #takes an opposite position in the contract two
            #expirations out from the front month
            self.order((sym, data[sym].expirations(2)), -5)

    Issues

  • This API has conflicts with the current roll implementation: Because roll determines front month size using the aggregates of the underlying, this position would result in a net zero position. Therefore on roll, both legs would be flattened and no order would be placed in the front month since there is a conditional preventing orders for zero contracts
